## 2. Implementation Work — Step by Step

### Step 1: Study the GCN Template

Before touching any file, I studied `models/gcn.py` carefully:

```python
class GCN(nn.Module):
    def __init__(self, in_channels, hidden_channels, out_channels,
                 num_layers=2, dropout=0.5, **kwargs):
        super().__init__()
        self.dropout = dropout
        self.convs = nn.ModuleList()
        self.convs.append(GCNConv(in_channels, hidden_channels))
        for _ in range(num_layers - 2):
            self.convs.append(GCNConv(hidden_channels, hidden_channels))
        self.convs.append(GCNConv(hidden_channels, out_channels))

    def forward(self, data):
        x, edge_index = data.x, data.edge_index
        for conv in self.convs[:-1]:
            x = conv(x, edge_index)
            x = F.relu(x)
            x = F.dropout(x, p=self.dropout, training=self.training)
        x = self.convs[-1](x, edge_index)   # final layer: no activation
        return x
```

Key design rules I extracted:
- `**kwargs` absorbs unused config keys (e.g. `pe_dim` for non-LSPE models)
- First N-1 layers: conv → ReLU → Dropout
- Last layer: conv only, raw logits — no activation (CrossEntropyLoss expects raw logits)
- `forward(data)` takes the full PyG data object

### Step 2: GraphSAGE (`models/graphsage.py`)

**Approach**: Direct translation of GCN, replace `GCNConv` with `SAGEConv`.

**Why SAGEConv works here**: PyG's `SAGEConv` uses mean aggregation by default and handles the concat-then-project internally. The interface is identical to `GCNConv` — takes `(x, edge_index)` and returns transformed features.

```python
from torch_geometric.nn import SAGEConv

class GraphSAGE(nn.Module):
    def __init__(self, in_channels, hidden_channels, out_channels,
                 num_layers=2, dropout=0.5, **kwargs):
        super().__init__()
        self.dropout = dropout
        self.convs = nn.ModuleList()
        self.convs.append(SAGEConv(in_channels, hidden_channels))
        for _ in range(num_layers - 2):
            self.convs.append(SAGEConv(hidden_channels, hidden_channels))
        self.convs.append(SAGEConv(hidden_channels, out_channels))

    def forward(self, data):
        x, edge_index = data.x, data.edge_index
        for conv in self.convs[:-1]:
            x = conv(x, edge_index)
            x = F.relu(x)
            x = F.dropout(x, p=self.dropout, training=self.training)
        x = self.convs[-1](x, edge_index)
        return x
```

**Verified result**: 79.7% test on Cora (seed=42).

### Step 3: GAT (`models/gat.py`)

**Approach**: Use `GATConv` with careful multi-head dimension management.

**The critical dimension problem**: If `hidden_channels=64` and `heads=4`, each head should output `64/4 = 16` features, and `concat=True` gives `4 × 16 = 64` total. This must be correct at every intermediate layer so dimensions chain correctly.

**Rules implemented**:
- `head_dim = hidden_channels // heads` (= 16 for heads=4, hidden=64)
- Intermediate layers: `GATConv(in_dim, head_dim, heads=heads, concat=True)` → output = `hidden_channels`
- Final layer: `GATConv(hidden_channels, out_channels, heads=1, concat=False)` → raw logits
- Activation: `F.elu` (standard for GAT per the original paper — not ReLU)
- Dropout applied to *input* of each layer (before conv), not to outputs

```python
class GAT(nn.Module):
    def __init__(self, in_channels, hidden_channels, out_channels,
                 num_layers=2, dropout=0.5, heads=4, **kwargs):
        super().__init__()
        assert hidden_channels % heads == 0
        head_dim = hidden_channels // heads

        self.convs = nn.ModuleList()
        if num_layers == 1:
            self.convs.append(GATConv(in_channels, out_channels, heads=1, concat=False, dropout=dropout))
        else:
            self.convs.append(GATConv(in_channels, head_dim, heads=heads, concat=True, dropout=dropout))
            for _ in range(num_layers - 2):
                self.convs.append(GATConv(hidden_channels, head_dim, heads=heads, concat=True, dropout=dropout))
            self.convs.append(GATConv(hidden_channels, out_channels, heads=1, concat=False, dropout=dropout))

    def forward(self, data):
        x, edge_index = data.x, data.edge_index
        for conv in self.convs[:-1]:
            x = F.dropout(x, p=self.dropout, training=self.training)
            x = conv(x, edge_index)
            x = F.elu(x)
        x = F.dropout(x, p=self.dropout, training=self.training)
        x = self.convs[-1](x, edge_index)
        return x
```

**Verified result**: 82.9% test on Cora (seed=42).

### Step 4: LSPE-GCN (`models/lspe.py`)

This was the main implementation challenge. I studied the paper carefully before writing any code.

#### Architecture Design Decisions

**Decision 1: Custom MessagePassing class vs built-in conv**

LSPE requires two separate aggregation passes per layer (one for `concat(h,p)`, one for `p` alone). There is no built-in PyG layer that does this. I implemented `LSPELayer` inheriting from `MessagePassing` with `aggr='mean'`:

```python
class LSPELayer(MessagePassing):
    def __init__(self, h_dim, pe_dim):
        super().__init__(aggr='mean')
        self.W_h = nn.Linear(h_dim + pe_dim, h_dim)
        self.W_p = nn.Linear(pe_dim, pe_dim)
        self.bn_h = nn.BatchNorm1d(h_dim)
        self.bn_p = nn.BatchNorm1d(pe_dim)
```

**Decision 2: Self-loops must be added manually**

The paper says MEAN over `N(v) ∪ {v}` (neighbourhood including the node itself). PyG's `aggr='mean'` does not add self-loops automatically — I had to call `add_self_loops(edge_index)` inside `forward()`:

```python
edge_index_sl, _ = add_self_loops(edge_index, num_nodes=h.size(0))
hp = torch.cat([h, p], dim=-1)
hp_agg = self.propagate(edge_index_sl, x=hp)   # h stream
p_agg  = self.propagate(edge_index_sl, x=p)    # p stream (separate call!)
```

**Decision 3: GCNConv input encoder (not plain Linear)**

Initially I used `nn.Linear(in_channels, hidden_channels)` for the input h encoding. This caused poor performance on Citeseer (58% vs GCN's 72%) because a plain linear layer compresses 3703 features to 128 with no graph structure at all.

I switched to `GCNConv(in_channels, hidden_channels)` which performs the compression while aggregating neighbour features — the same way all subsequent LSPE layers work. This improved Citeseer significantly.

**Decision 4: BatchNorm1d after every projection**

On Planetoid's tiny training sets, without BN the training loss collapsed to 0.007 within 50 epochs (the model perfectly memorised the 140 training labels). Val accuracy then oscillated wildly. BN normalises activations across all N nodes (not just training nodes), providing stable gradients throughout training.

**Decision 5: p NOT used in classifier**

Only `h^L` goes into the final `nn.Linear(hidden, num_classes)`. The paper is explicit about this — p serves only as enrichment for h during message passing.

#### Final LSPE forward pass

```python
def forward(self, data):
    # Step 1: Graph-aware input encoding
    h = F.relu(self.input_bn(self.input_conv(data.x, data.edge_index)))
    h = F.dropout(h, p=self.dropout, training=self.training)
    p = data.pe  # Laplacian eigenvectors, unchanged at input

    # Step 2: L LSPE layers, each updating both h and p
    for layer in self.layers:
        h, p = layer(h, p, data.edge_index)
        h = F.dropout(h, p=self.dropout, training=self.training)

    # Step 3: Classify using h only
    return self.classifier(h)
```

## 3. Hyperparameter Tuning

### Problem Encountered

After implementing LSPE, the initial config (`lr=0.005, epochs=500, dropout=0.5`) gave poor results. The training loss collapsed to 0.007 in 50 epochs but validation accuracy was ~70% and oscillating — a clear sign of overfitting on the 140-node training set.

### Tuning Process (Cora)

I ran a systematic grid search varying dropout, lr, weight_decay, and hidden_dim.

**Key finding**: `dropout=0.7 + weight_decay=0.005` is necessary to prevent memorisation. Standard values (do=0.5, wd=5e-4) are insufficient for a model with more parameters than GCN trained on 140 examples.

**Best Cora config**: `H=64, do=0.7, lr=0.01, wd=0.005, epochs=300` → val=76.4%, test=78.4%

### Citeseer-specific Tuning

Citeseer (3703 features, 120 training nodes) is structurally harder. The large input dimension means the GCNConv input encoder makes a very aggressive compression (3703→256). I tested `hidden_dim` up to 256 and found:

**Best Citeseer config**: `H=256, do=0.7, lr=0.01, wd=0.005, epochs=300` → val=64.0%, test=61.6%

This is lower than GCN's 72% — this is an architectural limitation, not a bug. See Section 4 below.

### PubMed-specific Tuning

PubMed has 500 features (smaller than Citeseer) but only 60 training nodes. Lower weight_decay worked better here.

**Best PubMed config**: `H=64, do=0.5, lr=0.01, wd=0.0005, epochs=300` → val=73.0%, test=70.8%

## 5. Final Verified Results

All 12 runs completed successfully, exit code 0 (seed=42, pe_dim=8, num_layers=2).

| Dataset | GCN | GraphSAGE | GAT | LSPE |
|---------|-----|-----------|-----|------|
| **Cora** | 82.4% | 79.7% | **82.9%** | 78.4% |
| **Citeseer** | **72.0%** | 70.3% | 71.7% | 61.6% |
| **PubMed** | **79.4%** | 77.6% | 77.5% | 70.8% |

**Why LSPE < GCN on Citeseer and PubMed**:

1. **Extra parameters**: LSPE has ~102K params vs GCN's ~92K. With 60–140 training nodes, this additional capacity is harder to regularise.
2. **Input encoding layer**: LSPE adds one extra GCNConv before the LSPE layers. Even with BN, this is one more layer's worth of parameters that the optimiser must tune with very few labels.
3. **Planetoid is not LSPE's target benchmark**: The paper showed LSPE improvements on ZINC, CLUSTER, TSP, and OGBG datasets — all larger with richer graph structures where Laplacian PE provides more useful signal. Cora/Citeseer/PubMed citation graphs have limited structural variety.

This is a valid and interesting finding for the project report: **the benefit of positional encodings depends on dataset scale and structural richness**.
